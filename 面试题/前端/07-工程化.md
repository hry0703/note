## 1. Webpack 核心概念与配置

**问：Webpack 的核心概念有哪些？**

**答：**

| 概念 | 说明 |
|------|------|
| **Entry** | 入口，从哪个文件开始打包 |
| **Output** | 输出，打包结果的位置和命名 |
| **Loader** | 处理非 JS 文件（如 CSS、图片、TS） |
| **Plugin** | 扩展功能（如 HtmlWebpackPlugin、压缩） |
| **Mode** | development / production |

```javascript
// webpack.config.js
module.exports = {
  entry: './src/index.js',
  output: {
    path: path.resolve(__dirname, 'dist'),
    filename: 'bundle.[contenthash].js',
  },
  module: {
    rules: [
      { test: /\.css$/, use: ['style-loader', 'css-loader'] },
      { test: /\.ts$/, use: 'ts-loader' },
    ],
  },
  plugins: [
    new HtmlWebpackPlugin({ template: './index.html' }),
  ],
  mode: 'production',
};
```

## 2. Vite 原理与优势

**问：Vite 为什么快？和 Webpack 的区别？**

**答：**

**原理**：
- **开发环境**：利用浏览器原生 ESM，不打包，按需请求。用 esbuild 预构建依赖（node_modules）
- **生产环境**：用 Rollup 打包

**优势**：
- 冷启动快：无需先打包整个项目
- HMR 快：只更新改动的模块
- 依赖预构建用 Go 写的 esbuild，速度极快

**与 Webpack 对比**：

| 对比 | Webpack | Vite |
|------|---------|------|
| 开发 | 先打包再启动 | 直接启动，按需编译 |
| 依赖处理 | 全部打包 | 预构建 + 强缓存 |
| 生产 | Webpack 打包 | Rollup 打包 |
| 生态 | 成熟 | 快速成长 |

## 3. Tree Shaking 原理

**问：什么是 Tree Shaking？如何生效？**

**答：**

**定义**：移除未使用的代码（死代码消除），减小 bundle 体积。

**前提**：
- 使用 **ES Module**（静态结构，可静态分析）
- 生产模式（`mode: 'production'`）
- 代码无副作用（或通过 `sideEffects` 声明）

```javascript
// utils.js
export const a = 1;
export const b = 2;

// main.js
import { a } from './utils';  // b 会被 tree-shaking 掉
```

**注意**：CommonJS 动态导入难以静态分析，Tree Shaking 效果有限。

## 4. 构建优化策略

**问：有哪些构建优化手段？**

**答：**

1. **代码分割**：`splitChunks`、动态 import，按路由或按需加载
2. **压缩**：Terser 压缩 JS，css-minimizer 压缩 CSS
3. **缓存**：`contenthash` 做文件名，利用浏览器缓存
4. **缩小范围**：`include/exclude` 减少 loader 处理范围
5. **多线程**：`thread-loader`、`esbuild-loader` 加速
6. **DLL**：预编译不常变的依赖，减少重复构建
7. **分析**：`webpack-bundle-analyzer` 分析体积，针对性优化

```javascript
optimization: {
  splitChunks: {
    chunks: 'all',
    cacheGroups: {
      vendor: {
        test: /[\\/]node_modules[\\/]/,
        name: 'vendors',
      },
    },
  },
},
```

## 5. Monorepo

**问：什么是 Monorepo？有什么工具？**

**答：**

**定义**：多个项目/包放在一个仓库中，共享依赖和配置。

**优势**：代码复用、统一版本、原子化提交、便于重构。

**工具**：
- **pnpm workspace**：轻量，依赖管理好
- **Lerna**：老牌，适合发 npm 包
- **Turborepo**：增量构建、缓存、任务编排
- **Nx**：功能全面，适合大型 Monorepo

```json
// pnpm-workspace.yaml
packages:
  - 'packages/*'
  - 'apps/*'
```

## 6. 微前端架构

**问：微前端是什么？有哪些方案？**

**答：**

**定义**：将大型应用拆成多个可独立开发、部署、运行的前端应用，再组合成整体。

**常见方案**：

| 方案 | 原理 | 特点 |
|------|------|------|
| **qiankun** | 基于 single-spa，JS 沙箱 + 样式隔离 | 接入简单，应用广泛 |
| **Module Federation** | Webpack 5 模块联邦 | 运行时共享依赖 |
| **iframe** | 子应用用 iframe 嵌入 | 隔离强，通信和体验差 |
| **EMP** | 基于 Module Federation | 支持多框架 |

**核心问题**：样式隔离、JS 隔离、应用通信、路由同步、公共依赖共享。
