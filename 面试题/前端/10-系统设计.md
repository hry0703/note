## 1. 监控 SDK 设计

**问：如何设计一个前端监控 SDK？**

**答：**

**核心能力**：
- **错误监控**：JS 错误、Promise 未捕获、资源加载失败
- **性能监控**：首屏时间、接口耗时、Core Web Vitals
- **行为监控**：点击、路由、接口请求（可选）

**设计要点**：

1. **错误捕获**：`window.onerror`、`unhandledrejection`、`addEventListener('error')`（资源）
2. **数据上报**：`sendBeacon`（可靠）、`img`（简单）、批量上报（减少请求）
3. **采样**：按比例采样，控制数据量
4. **防重复**：同一错误去重、节流
5. **离线缓存**：失败时存 localStorage，下次补发
6. **轻量**：SDK 体积小，异步加载，不阻塞主流程

```javascript
// 简单错误上报
window.onerror = (msg, url, line, col, error) => {
  report({ type: 'error', msg, url, line, col, stack: error?.stack });
};
window.addEventListener('unhandledrejection', (e) => {
  report({ type: 'unhandledrejection', reason: e.reason });
});
```

## 2. 断点续传实现

**问：大文件如何实现断点续传？**

**答：**

**思路**：文件分片 → 每片单独上传 → 记录已传分片 → 失败/刷新后从未传分片继续。

**步骤**：

1. **分片**：`File.slice(start, end)` 切成固定大小（如 2MB）
2. **计算 hash**：用 SparkMD5 等计算文件/分片 hash，用于秒传和校验
3. **上传**：每片单独请求，带 `hash`、`index`、`total`
4. **服务端**：按 hash 建临时目录，接收分片，全部到齐后合并
5. **续传**：上传前先请求「已上传分片列表」，跳过已传

```javascript
const chunkSize = 2 * 1024 * 1024;
const chunks = Math.ceil(file.size / chunkSize);

for (let i = 0; i < chunks; i++) {
  const chunk = file.slice(i * chunkSize, (i + 1) * chunkSize);
  await uploadChunk(chunk, i, chunks, fileHash);
}
```

## 3. 秒杀系统前端设计

**问：秒杀场景下前端如何设计？**

**答：**

**前端策略**：

1. **防重复提交**：按钮点击后禁用、防抖，提交中禁止再次点击
2. **本地排队**：请求失败或排队时，前端轮询或 WebSocket 获取状态
3. **预加载**：提前加载静态资源、预连接接口域名
4. **降级**：接口超时或失败时，提示排队中、稍后重试
5. **限流提示**：服务端限流时，前端友好提示、引导错峰
6. **静态化**：秒杀页尽量静态，减少接口依赖

**配合后端**：限流、队列、库存扣减、防超卖等由后端保证。

## 4. 大文件上传

**问：如何实现大文件上传？**

**答：**

**方案一：分片上传**（同断点续传）
- 分片、并发上传、断点续传、秒传（hash 校验）

**方案二：Web Worker + 分片**
- 在 Worker 中计算 hash、分片，不阻塞主线程

**方案三：压缩**
- 图片压缩后上传，减小体积

**关键点**：
- 分片大小：2–5MB 较常见
- 并发数：3–8 个，避免过多连接
- 进度：监听 `xhr.upload.onprogress` 或 `axios` 的 `onUploadProgress`
- 秒传：先传 hash，服务端已有则直接返回成功
